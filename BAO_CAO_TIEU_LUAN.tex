\section{Kiến Trúc Microservices và Domain-Driven Design}

\subsection{Từ Monolithic đến Microservices}

Kiến trúc phần mềm truyền thống (Monolithic Architecture) gom tất cả chức năng vào một ứng dụng duy nhất, khó bảo trì, khó mở rộng. Khi hệ thống có hàng trăm nghiệp vụ và hàng tỉ giao tác, một số vấn đề nổi cộm:

\begin{itemize}
    \item \textbf{Kết dính chặt chẽ (Tight Coupling)}: Thay đổi một tính năng dễ ảnh hưởng toàn hệ.
    \item \textbf{Khó mở rộng (Poor Scalability)}: Phải duplicate toàn bộ ứng dụng.
    \item \textbf{Khó triển khai (Deployment Challenge)}: Một lỗi nhỏ làm crash toàn bộ.
    \item \textbf{Khó bảo trì (Maintenance Burden)}: Codebase khổng lồ, khó hiểu.
\end{itemize}

\textbf{Giải pháp:} Kiến trúc Microservices (vi dịch vụ) tách ứng dụng thành các dịch vụ nhỏ, độc lập, giao tiếp qua API. Mỗi dịch vụ:
\begin{itemize}
    \item Quản lý một phần nghiệp vụ riêng biệt (Single Responsibility Principle).
    \item Có cơ sở dữ liệu riêng (Database Per Service Pattern).
    \item Phi tập trung, có thể triển khai độc lập.
    \item Cho phép sử dụng công nghệ khác nhau (polyglot architecture).
\end{itemize}

\textbf{Ưu điểm của Microservices:}
\begin{enumerate}
    \item \textbf{Khả năng mở rộng cao} – Scale từng service riêng lẻ theo load.
    \item \textbf{Khả năng phục hồi} – Một service lỗi không làm crash toàn hệ.
    \item \textbf{Linh hoạt công nghệ} – Mỗi service chọn tech stack tối ưu.
    \item \textbf{Dễ bảo trì} – Team nhỏ quản lý một service cụ thể.
    \item \textbf{Tốc độ phát triển} – Deploy nhanh mà không cần release toàn hệ.
\end{enumerate}

\textbf{Thách thức của Microservices:}
\begin{enumerate}
    \item \textbf{Độ phức tạp giao tiếp} – Phải quản lý distributed calls, timeout, retry.
    \item \textbf{Tính nhất quán dữ liệu} – Phải kết hợp dữ liệu từ nhiều database.
    \item \textbf{Logging và Monitoring} – Cần centralized tracing để debug.
    \item \textbf{Deployment} – Cần orchestration tool (Docker, Kubernetes).
\end{enumerate}

\subsection{Thiết Kế Microservices Theo Domain-Driven Design}

\textbf{Domain-Driven Design (DDD)} là phương pháp thiết kế phần mềm bằng cách mô hình hóa các nghiệp vụ thực tế (domain). Thay vì tập trung vào kỹ thuật (framework, database), DDD tập trung vào \textit{ngôn ngữ chung} giữa kỹ sư và chuyên gia kinh doanh.

\textbf{Các khái niệm cốt lõi của DDD:}

\begin{description}
    \item[Ubiquitous Language (Ngôn ngữ chung)] Dùng từ vựng thống nhất để nói về domain trong code, documentation, và hội thoại.
    
    \item[Bounded Context (Ngữ cảnh giới hạn)] Chia domain lớn thành các phần nhỏ, mỗi phần có:
    \begin{itemize}
        \item Ngôn ngữ riêng
        \item Model riêng
        \item Ranh giới rõ ràng
        \item Giao tiếp qua API/Events
    \end{itemize}
    
    \item[Aggregate (Tập hợp)] Nhóm các entities liên quan thành một đơn vị quản lý trans actional. Chỉ có thể modify thông qua aggregate root.
    
    \item[Entity (Thực thể)] Đối tượng có identity không đổi. Ví dụ: Patient (bệnh nhân) có ID không đổi ngay cả khi thay đổi tên/địa chỉ.
    
    \item[Value Object (Đối tượng giá trị)] Không có identity riêng, chỉ được định nghĩa bởi giá trị. Ví dụ: Amount (số tiền), Priority (mức độ ưu tiên). Không thể thay đổi – phải tạo mới.
    
    \item[Domain Event (Sự kiện miền)] Ghi lại sự kiện quan trọng xảy ra trong domain. Các dịch vụ khác lắng nghe để sync dữ liệu. Ví dụ: \textit{PatientRegistered}, \textit{PrescriptionSubmitted}.
    
    \item[Repository (Kho lưu trữ)] Abstraction để truy cập data. Che dấu chi tiết về cơ sở dữ liệu (SQL, NoSQL, v.v.).
    
    \item[Domain Service (Dịch vụ miền)] Logic nghiệp vụ không thuộc entity/value object. Ví dụ: RecommendationEngine (tính toán gợi ý).
\end{description}

\textbf{Kiến trúc 4-Layer DDD:}

Mỗi Bounded Context (microservice) được tổ chức thành 4 lớp:

\begin{enumerate}
    \item \textbf{Interface Layer (Lớp giao diện)} – REST API, HTTP endpoints, nhận request từ client.
    
    \item \textbf{Application Layer (Lớp ứng dụng)} – CQRS (Commands \& Queries). Command (viết), Query (đọc). Orchestrate domain logic.
    
    \item \textbf{Domain Layer (Lớp miền)} – Aggregates, Entities, Value Objects, Domain Services, Domain Events. Chứa logic nghiệp vụ.
    
    \item \textbf{Infrastructure Layer (Lớp cơ sở hạ tầng)} – Database, Repository Implementation, External Service Calls.
\end{enumerate}

\begin{verbatim}
******* DDD 4-LAYER ARCHITECTURE *******

[1] INTERFACE LAYER (REST API)
    └─> GET /patients, POST /consultations
    
[2] APPLICATION LAYER (CQRS Pattern)
    ├─ Commands: CreatePatientCommand
    └─ Queries: ListPatientsQuery
    
[3] DOMAIN LAYER (Business Logic)
    ├─ Patient Aggregate (root entity)
    └─ Domain Events: PatientCreated
    
[4] INFRASTRUCTURE LAYER (Database)
    ├─ PatientRepository
    └─ PostgreSQL + SQLite ORM
    
*****  Data flows from top to bottom  *****
\end{verbatim}

Lợi ích của kiến trúc này:
\begin{itemize}
    \item \textbf{Độc lập kiểm thử} – Test domain logic không cần database.
    \item \textbf{Dễ bảo trì} – Mỗi lớp có trách nhiệm rõ.
    \item \textbf{Linh hoạt công nghệ} – Swap database dễ dàng.
    \item \textbf{Reusable} – Domain logic không phụ thuộc framework.
\end{itemize}

\subsection{Case Study: Hệ Thống E-Commerce Healthcare}

\textbf{Bài toán đặt vấn đề:}

Trong bối cảnh phát triển của ngành y tế điện tử (E-health), người dùng Việt Nam gặp những khó khăn:

\begin{enumerate}
    \item \textbf{Thông tin sức khỏe phân tán} – Bệnh nhân phải gửi lại hồ sơ ở mỗi cơ sở y tế.
    
    \item \textbf{Tìm kiếm tư vấn y tế khó} – Thông tin trên Internet rối ren, không chuyên môn, dễ gây hoang mang.
    
    \item \textbf{Tra cứu thuốc phức tạp} – Không biết thuốc nào phù hợp, tương tác với thuốc khác.
    
    \item \textbf{Gợi ý cạnh tranh} – Khó biết sản phẩm nào phù hợp với tình trạng sức khỏe riêng.
    
    \item \textbf{Đơn thuốc không được quản lý} – Không có lịch sử đơn thuốc, không cảnh báo tương tác.
\end{enumerate}

\textbf{Giải pháp: Nền tảng Health-Micro-AI}

Dự án này xây dựng một nền tảng e-commerce y tế hoàn chỉnh với:

\begin{itemize}
    \item \textbf{Quản lý bệnh nhân tập trung} – Hồ sơ y tế điện tử (EHR – Electronic Health Record).
    \item \textbf{Chatbot AI tư vấn y tế} – Sử dụng GraphRAG (Neo4j + FAISS + LLM Gemini) để trả lời câu hỏi y tế dựa trên kiến thức khoa học.
    \item \textbf{Gợi ý cá nhân hóa} – Sử dụng Collaborative Filtering và GNN để gợi ý sản phẩm phù hợp.
    \item \textbf{Quản lý đơn thuốc} – Tạo, lưu giữ, và theo dõi đơn thuốc.
    \item \textbf{Danh mục y khoa} – Theo ICD-10, phân loại bệnh tật và sản phẩm.
    \item \textbf{Đánh giá cộng đồng} – Người dùng đánh giá hiệu quả điều trị, chia sẻ kinh nghiệm.
\end{itemize}

Bằng cách sử dụng Microservices + DDD, hệ thống này:
\begin{itemize}
    \item Cho phép mở rộng riêng từng chức năng (ví dụ: scale AI service khi tầng lạnh tăng).
    \item Bảo vệ từng miền nghiệp vụ bằng Bounded Context.
    \item Giảm độ phức tạp và tăng khả năng bảo trì.
\end{itemize}

\section{Hệ Thống E-Commerce Healthcare: Yêu Cầu và Phân Rã}

\subsection{Xác Định Yêu Cầu Chức Năng}

\textbf{Yêu cầu chức năng (Functional Requirements):}

Hệ thống phải hỗ trợ các vai trò:
\begin{enumerate}
    \item \textbf{Bệnh nhân (Customer)} – Tìm kiếm, mua thuốc, tư vấn AI, quản lý đơn thuốc.
    
    \item \textbf{Quản trị viên (Admin)} – Quản lý sản phẩm, danh mục, người dùng, duyệt đơn thuốc.
    
    \item \textbf{Nhân viên (Staff)} – Xử lý đơn thuốc, kiểm kho, cập nhật tồn kho.
\end{enumerate}

\textbf{Chức năng nòng cốt:}
\begin{itemize}
    \item \textbf{Xác thực và phân quyền} – Đăng ký, đăng nhập (JWT), phân vai trò.
    
    \item \textbf{Quản lý bệnh nhân} – Hồ sơ y tế, thông tin cá nhân, lịch khám.
    
    \item \textbf{Quản lý sản phẩm} – Danh mục thuốc, tìm kiếm, filter theo triệu chứng.
    
    \item \textbf{Chatbot AI tư vấn} – Trả lời câu hỏi y tế, gợi ý sản phẩm.
    
    \item \textbf{Gợi ý cá nhân hóa} – Recommend sản phẩm dựa trên hành vi và sở thích.
    
    \item \textbf{Quản lý đơn thuốc} – Tạo, lưu trữ, theo dõi trạng thái đơn.
    
    \item \textbf{Đánh giá và bình luận} – Người dùng đánh giá sản phẩm, chia sẻ kinh nghiệm.
\end{itemize}

\textbf{Yêu cầu phi chức năng (Non-Functional Requirements):}
\begin{itemize}
    \item \textbf{Khả năng mở rộng} – Hỗ trợ hàng triệu người dùng, mười triệu giao tác/ngày.
    
    \item \textbf{Độ tin cậy} – Uptime $\geq 99.9\%$ (SLA 3-nines).
    
    \item \textbf{Hiệu suất} – Response time $< 200$ms cho hầu hết request.
    
    \item \textbf{Bảo mật} – Bảo vệ thông tin bệnh nhân (HIPA, GDPR-like compliance).
    
    \item \textbf{Tính nhất quán dữ liệu} – Sử dụng event-driven architecture để đảm bảo consistency.
\end{itemize}

\subsection{Phân Rã Hệ Thống Theo DDD}

Bằng cách phân tích domain, ta xác định được các \textbf{Bounded Contexts} (ngữ cảnh giới hạn):

\begin{enumerate}
    \item \textbf{User Management Context} – Quản lý tài khoản, xác thực, phân quyền.
    
    \item \textbf{Patient Management Context} – Hồ sơ y tế, thông tin cá nhân bệnh nhân.
    
    \item \textbf{Product Catalog Context} – Danh mục sản phẩm, tìm kiếm, phân loại.
    
    \item \textbf{Prescription Management Context} – Đơn thuốc, giỏ hàng tạm thời.
    
    \item \textbf{AI Consulting Context} – Chatbot GraphRAG, tư vấn y tế.
    
    \item \textbf{Recommendation Context} – Gợi ý sản phẩm, machine learning.
    
    \item \textbf{Review \& Rating Context} – Đánh giá, bình luận sản phẩm.
\end{enumerate}

\subsection{Từng Service – Một Trong Các Bounded Contexts}

\subsubsection{1. Auth Service (Port :8012, FastAPI)}

\textbf{Mục tiêu:} Quản lý xác thực (authentication) và phân quyền (authorization).

\textbf{Chức năng:}
\begin{itemize}
    \item Đăng ký tài khoản mới.
    \item Đăng nhập, cấp JWT token (access\_token, refresh\_token).
    \item Xác thực token (verify, refresh).
    \item Quản lý vai trò (Role), quyền (Permission).
    \item Quản lý user (lock account, change password).
\end{itemize}

\textbf{Domain Model (DDD):}
\begin{itemize}
    \item \textbf{Entities:} User (aggregate root), Role, Permission.
    \item \textbf{Value Objects:} UserId, Email, CredentialHash, Token, JWTClaims.
    \item \textbf{Domain Events:} UserRegistered, UserLoggedIn, TokenGenerated, PasswordChanged, RoleAssigned.
    \item \textbf{Domain Services:} AuthenticationService, AuthorizationService, TokenManagementService.
\end{itemize}

\section{Dịch Vụ AI: Chatbot GraphRAG và Gợi Ý Cá Nhân Hóa}

\subsection{Cơ Sở Lý Thuyết}

\subsubsection{Retrieval-Augmented Generation (RAG)}

\textbf{Định nghĩa:} RAG là kĩ thuật kết hợp \textit{Information Retrieval} (tìm kiếm thông tin) với \textit{Generative AI} (sinh văn bản). Thay vì chỉ dùng LLM để sinh câu trả lời (có nguy cơ hallucination – tạo ra thông tin sai), RAG:

\begin{enumerate}
    \item Truy xuất (retrieve) các tài liệu liên quan từ knowledge base.
    \item Tăng cường (augment) prompt bằng các tài liệu này.
    \item Yêu cầu LLM sinh câu trả lời dựa trên tài liệu (grounded).
\end{enumerate}

\textbf{Công thức toán:}
$$P(\text{answer}|\text{question}, \text{retrieved\_docs}) \propto P(\text{docs}|\text{question}) \cdot P(\text{answer}|\text{docs}, \text{question})$$

Trong đó:
\begin{itemize}
    \item $P(\text{docs}|\text{question})$ là xác suất tài liệu liên quan (retrieval stage).
    \item $P(\text{answer}|\text{docs}, \text{question})$ là xác suất câu trả lời cho trước tài liệu (generation stage).
\end{itemize}

\textbf{Lợi ích của RAG:}
\begin{itemize}
    \item \textbf{Giảm hallucination} – LLM phải trích dẫn từ tài liệu thực tế.
    \item \textbf{Dễ cập nhật} – Chỉ cần thêm tài liệu mới, không cần retraining model.
    \item \textbf{Transparent} – Có thể theo dõi nguồn thông tin.
    \item \textbf{Specificity} – Câu trả lời phù hợp domain (y tế, pháp luật, v.v.).
\end{itemize}

\subsubsection{Graph Retrieval Augmented Generation (GraphRAG)}

\textbf{Nâng cấp RAG:} Kết hợp \textbf{Knowledge Graph} (đồ thị tri thức) với Vector Store.

\begin{itemize}
    \item \textbf{Knowledge Graph (Neo4j)} – Biểu diễn mối quan hệ có cấu trúc:
    \begin{verbatim}
(Symptom: "Đau dạ dày")-[:INDICATES]->(Disease: "Gastric Ulcer")
(Disease: "Gastric Ulcer")-[:TREATED_BY]->(Product: "Omeprazole")
(Product: "Omeprazole")-[:CATEGORY]->(Category: "PPI")
(User: "Patient123")-[:SEARCHED]->(Symptom: "Đau dạ dày")
    \end{verbatim}
    Lợi ích: Tìm kiếm theo logic quan hệ, không chỉ semantic similarity.
    
    \item \textbf{Vector Store (FAISS)} – Tìm kiếm phi cấu trúc dựa mức độ tương đồng vector:
    \begin{verbatim}
Query: "Tôi bị đau bụng vào buổi sáng sớm"
→ Vector encode → FAISS search top-5 similar chunks
→ Chunks về GERD, Peptic Ulcer, IBS, ...
    \end{verbatim}
    Lợi ích: Semantic search, catch queries không match exact keywords.
\end{itemize}

\textbf{Combine 3 nguồn:}
\begin{enumerate}
    \item Neo4j Knowledge Graph – Truy xuất facts có cấu trúc.
    \item FAISS Vector Store – Truy xuất docs phi cấu trúc.
    \item Product Service REST API – Fetch thông tin sản phẩm thực tế.
\end{enumerate}

\subsubsection{Phân Tích Hành Vi Khách Hàng (Customer Behavior Analysis)}

\textbf{Mục đích:} Hiểu nhu cầu, sở thích của khách hàng để gợi ý phù hợp.

\textbf{Kỹ thuật:}
\begin{itemize}
    \item \textbf{Collaborative Filtering} – Assume: người dùng có hành vi tương tự → nên like sản phẩm tương tự.
    \[ \text{similarity}(u, v) = \frac{\vec{u} \cdot \vec{v}}{|\vec{u}| |\vec{v}|} \]
    
    \item \textbf{Content-Based Filtering} – Recommend dựa trên sản phẩm mà user đã like.
    
    \item \textbf{Hybrid} – Kết hợp cả hai.
\end{itemize}

\textbf{Trong Health-Micro-AI:}
\begin{itemize}
    \item Ghi nhận hành vi: product viewed, products searched, products purchased.
    \item Neo4j node: (User)-[:VIEWED {weight}]->(Product), (User)-[:SEARCHED]->(Symptom).
    \item Query: "Những người xem sản phẩm X cũng xem sản phẩm Y" → recommend Y.
\end{itemize}

\subsubsection{Graph Neural Networks (GNN) và SPD Manifold}

\textbf{Graph Neural Networks:} Mô hình học sâu tác động trên đồ thị. Mỗi node học biểu diễn (embedding) kết hợp thông tin từ neighbors.

\begin{itemize}
    \item \textbf{GraphSAGE} – Sample neighbors, aggregate their embeddings.
    \item \textbf{GAT} – Attention mechanism, trọng số cạnh động.
    \item \textbf{GCN} – Graph Convolutional Networks.
\end{itemize}

\textbf{SPD Manifold} (Symmetric Positive Definite): Không gian toán học đặc biệt dùng để bảo toàn cấu trúc không gian khi learn embeddings. Giúp recommendation không bị "collapse" vào vài chiều.

\textbf{Ứng dụng:} Dự đoán liên kết (link prediction) – "Người dùng này nên được recommend sản phẩm nào?" dựa trên cấu trúc đồ thị.

\subsubsection{Học Sâu (Deep Learning)}

\textbf{Định nghĩa:} Mạng neural nhiều lớp học biểu diễn phân cấp từ dữ liệu thô.

\textbf{Kiến trúc phổ biến:}
\begin{itemize}
    \item \textbf{Convolutional Neural Networks (CNN)} – Xử lý ảnh, temporal data.
    \item \textbf{Recurrent Neural Networks (RNN/LSTM)} – Xử lý sequence, time-series.
    \item \textbf{Transformer} – Self-attention, machine translation, LLM.
    \item \textbf{Graph Neural Networks (GNN)} – Xử lý đồ thị.
\end{itemize}

\textbf{Trong project:} Không train Deep Learning model locally. Thay vào đó:
\begin{itemize}
    \item Dùng Gemini API (pre-trained LLM) để embed text và sinh text.
    \item Dùng pre-trained model (SentenceTransformer) để embed medical documents.
    \item GNN dùng thư viện DGL hoặc PyTorch Geometric (optional, cho production).
\end{itemize}

\textbf{Định nghĩa:} Tập hợp thông tin khoa học được cấu trúc để các hệ thống AI có thể query.

\subsubsection{Recommender-AI Service: Collaborative Filtering}

\textbf{Chức năng:} Gợi ý sản phẩm dựa trên hành vi người dùng tương tự.

\textbf{Thuật toán Jaccard Similarity:}
$$J(u, v) = \frac{|I_u \cap I_v|}{|I_u \cup I_v|}$$

Trong đó $I_u$, $I_v$ là tập sản phẩm mà user $u$, $v$ đã view/purchase.

\textbf{Luồng:}
\begin{enumerate}
    \item Xây dựng user-item matrix: mỗi hàng là user, cột là product, giá trị là rating/weight.
    
    \item Tính similarity giữa current user và tất cả users khác dùng Jaccard.
    
    \item Lấy top-N similar users.
    
    \item Gợi ý products mà similar users viewed/purchased nhưng current user chưa.
    
    \item Rank theo confidence score (số similar users → product).
\end{enumerate}

\textbf{Pseudocode:}
\begin{verbatim}
def collaborative_filtering_recommend(user_id, k=5, n=10):
    user_items = get_user_items(user_id)  # set of product IDs
    
    similar_users = []
    for other_user in all_users:
        if other_user == user_id:
            continue
        other_items = get_user_items(other_user)
        similarity = jaccard_similarity(user_items, other_items)
        if similarity > 0:
            similar_users.append((other_user, similarity))
    
    similar_users.sort(by=similarity, desc=True)
    top_k_similar = similar_users[:k]
    
    candidates = {}  # product_id -> score
    for other_user, sim_score in top_k_similar:
        for product in get_user_items(other_user):
            if product not in user_items:
                candidates[product] += sim_score
    
    top_n_recommend = sorted(candidates.items(), 
                             by=score, desc=True)[:n]
    return top_n_recommend
\end{verbatim}

\begin{itemize}

    \item \textbf{Sparse data} – Nếu users ít, khó tính similarity.
\end{itemize}

class GraphRAGEngine:

        genai.configure(api_key=gemini_api_key)
        self.model = genai.GenerativeModel('gemini-1.5-flash')
        
        # Load FAISS index
        self.faiss_index = faiss.read_index(
            f"{faiss_index_path}/medical.index")
        with open(f"{faiss_index_path}/chunks.json") as f:
            self.chunks = json.load(f)
    
    def query_graph(self, symptom: str) -> List[Dict]:
        """Query Neo4j for symptom -> disease -> product relationships"""
        with self.driver.session() as session:
            result = session.run("""
                MATCH (s:Symptom{name: $symptom})
                      -[r:INDICATES]->
                      (d:Disease)
                      -[t:TREATED_BY]->
                      (p:Product)
                RETURN d.name as disease, 
                       d.icd_code, 
                       p.product_id,
                       p.name as product_name,
                       t.priority
                ORDER BY t.priority DESC
                LIMIT 10
            """, symptom=symptom)
            return [dict(record) for record in result]
    
    def search_vectors(self, query: str, k: int = 5) -> List[str]:
        """Search similar medical documents using FAISS"""
        # Embed query using Gemini
        response = genai.embed_content(
            model="models/embedding-001",
            content=query,
            task_type="RETRIEVAL_QUERY"
        )
        query_vector = np.array(response['embedding'], dtype=np.float32)
        query_vector = query_vector / np.linalg.norm(query_vector)
        
        # FAISS search
        distances, indices = self.faiss_index.search(
            np.array([query_vector]), k=k)
        
        results = []
        for idx in indices[0]:
            if idx < len(self.chunks):
                results.append(self.chunks[idx])
        return results
    
    def generate_answer(self, question: str, 
                       graph_context: List[Dict],
                       vector_context: List[str]) -> str:

        # Build context

MEDICAL DOCUMENTS:
{chr(10).join(vector_context)}

Provide a helpful, evidence-based answer.

Always add a medical disclaimer.

        response = self.model.generate_content(context)
        return response.text

\subsubsection{2. Collaborative Filtering Recommendation}

\begin{lstlisting}[language=Python, caption=Collaborative Filtering (recommender-ai-service/app.py)]
from collections import defaultdict
import json

class CollaborativeFilteringEngine:
    def __init__(self):
        self.user_items = {}  # user_id -> set of product_ids
    
    def jaccard_similarity(self, set1, set2):
        """Calculate Jaccard similarity between two sets"""
        if not set1 or not set2:
            return 0.0
        intersection = len(set1 & set2)
        union = len(set1 | set2)
        return intersection / union if union > 0 else 0.0
    
    def get_recommendations(self, user_id, 
                           k_similar_users=5, 
                           n_recommendations=10):
        """
        Recommend products to user based on similar users
        
        Args:
            user_id: Current user
            k_similar_users: Top-K similar users
            n_recommendations: Number of products to recommend
            
        Returns:
            List of (product_id, confidence_score) tuples
        """
        
        if user_id not in self.user_items:
            return []  # No history for this user
        
        user_items = self.user_items[user_id]
        
        # Calculate similarity with all other users
        similarities = []
        for other_user, other_items in self.user_items.items():
            if other_user == user_id:
                continue
            
            sim = self.jaccard_similarity(user_items, 
                                         set(other_items))
            if sim > 0:
                similarities.append((other_user, sim))
        
        # Get top-K similar users
        similarities.sort(key=lambda x: x[1], reverse=True)
        top_similar = similarities[:k_similar_users]
        
        # Aggregate candidate products
        candidate_scores = defaultdict(float)
        for similar_user, sim_score in top_similar:
            for product in self.user_items.get(similar_user, []):
                if product not in user_items:
                    candidate_scores[product] += sim_score
        
        # Sort and return top-N
        recommendations = sorted(candidate_scores.items(),
                                key=lambda x: x[1], 
                                reverse=True)[:n_recommendations]
        
        return recommendations
    
    def record_interaction(self, user_id, product_id, 
                          interaction_type="view"):
        """Record user-product interaction"""
        if user_id not in self.user_items:
            self.user_items[user_id] = []
        
        if product_id not in self.user_items[user_id]:
            self.user_items[user_id].append(product_id)
\end{lstlisting}

\begin{lstlisting}[language=Python, caption=API Gateway Routing (api-gateway/app/views.py)]
from django.http import JsonResponse
from django.views import View
import httpx
import json

class GatewayProxyView(View):
    """Proxy requests to microservices"""
    
    SERVICE_REGISTRY = {
        'auth': 'http://auth-service:8000',
        'patient': 'http://patient-service:8000',
        'pharmacy': 'http://pharmacy-service:8000',
        'prescription': 'http://prescription-service:8000',
        'clinical-advisory': 'http://clinical-advisory-service:8000',
        'recommender': 'http://recommender-ai-service:8000',
    }
    
    def route_request(self, request, service, path):
        """Route HTTP request to appropriate service"""
        
        # Verify JWT token
        auth_header = request.headers.get('Authorization', '')
        if not auth_header.startswith('Bearer '):
            return JsonResponse(
                {'error': 'Unauthorized'}, 
                status=401
            )
        
        # Get service URL
        if service not in self.SERVICE_REGISTRY:
            return JsonResponse(
                {'error': f'Unknown service: {service}'}, 
                status=404
            )
        
        service_url = self.SERVICE_REGISTRY[service]
        
        # Build target URL
        target_url = f"{service_url}/api/{path}"
        
        # Forward request using httpx
        try:
            with httpx.Client(timeout=10.0) as client:
                # Copy headers, preserve JWT
                headers = dict(request.headers)
                
                if request.method == 'GET':
                    response = client.get(target_url, headers=headers)
                elif request.method == 'POST':
                    response = client.post(
                        target_url, 
                        headers=headers,
                        content=request.body
                    )
                elif request.method == 'PUT':
                    response = client.put(
                        target_url, 
                        headers=headers,
                        content=request.body
                    )
                else:
                    return JsonResponse(
                        {'error': 'Method not allowed'}, 
                        status=405
                    )
                
                # Return upstream response
                return JsonResponse(
                    response.json(), 
                    status=response.status_code
                )
        
        except httpx.TimeoutException:
            return JsonResponse(
                {'error': 'Service timeout'}, 
                status=504
            )
        except Exception as e:
            return JsonResponse(
                {'error': str(e)}, 
                status=500
            )

    \item \textbf{Khả năng phục hồi} – Lỗi một service không làm crash toàn hệ.

    \item \textbf{Gợi ý cá nhân hóa} – Recommender Services cung cấp sản phẩm phù hợp cho từng người dùng.
\end{enumerate}
